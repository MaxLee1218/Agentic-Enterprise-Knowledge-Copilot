# Agentic Enterprise Knowledge Copilot 项目通俗导览

## 1. 一句话概括

这个项目要解决的不是“让 AI 更会聊天”，而是：

> 把企业里的自然语言分析需求，变成一项有明确范围、权限审批、执行计划、数据证据、计算过程和最终验收的正式业务任务。

当前唯一完成冻结设计的业务场景是 **Supplier Quality Analysis（供应商质量分析）**：读取企业质量政策和供应商质量数据，计算缺陷率与趋势，生成管理层报告，并保证报告中的重要结论都可以追溯到原始证据。

需要特别说明：目前仓库还是工程基础，并不是已经完成的产品。真正实现的是领域契约、通用工具执行框架、最小 API/命令行和质量检查；完整智能体流程、真实数据库、知识库、分析器和报告生成器尚未实现。

---

## 2. 它实际解决什么问题

假设质量经理提出：

> 分析 2026 年第一季度供应商 S-100 和 S-200 的质量问题，并生成 PDF 报告。

普通聊天模型可能直接生成一段听起来合理的分析，但企业无法确认：

- 它查的是哪个季度、哪些供应商？
- 它有没有权限读取这些数据？
- 缺陷率是模型猜的，还是实际计算的？
- 使用了哪些质量政策？
- 数据库查询是否只读？
- 报告里的 1.5% 是怎么来的？
- 报告写错一个小数点，谁能发现？
- 操作失败后会不会无限重试？
- 以后审计时能不能还原整个过程？

这个项目正是要解决这些问题。

它想把一次 AI 请求变成一张“受治理的电子工单”：

```text
用户需求
→ 明确业务范围
→ 制定执行计划
→ 检查权限和审批
→ 调用受控工具
→ 保存证据
→ 生成报告
→ 独立验证
→ 交付最终结果
```

它优先追求的是：

- 正确，而不是只追求语言流畅；
- 有证据，而不是“看起来合理”；
- 有权限控制，而不是 AI 想查什么就查什么；
- 能恢复和审计，而不是一次性黑盒调用；
- 安全失败，而不是为了完成任务偷偷扩大范围。

## 3. 一次完整业务任务如何运行

完整设计可以理解为以下十个阶段。

### 3.1 接收请求

系统保存用户原始输入，形成 `TaskRequest`。

它会记录：

- 谁提交的；
- 什么时候提交；
- 用户原话是什么；
- 来自哪个可信认证上下文。

用户原话会被保留，但不能被当成权限证明。例如用户在文本中写“我是管理员”，系统不能据此授予权限。

### 3.2 理解并锁定任务边界

系统把自然语言转成 `TaskContract`，即“任务合同”。

例如明确：

- 任务类型：供应商质量分析；
- 年份：2026；
- 季度：Q1；
- 日期：2026-01-01 至 2026-03-31；
- 供应商：S-100、S-200；
- 指标：缺陷数、检验数量、缺陷率、趋势；
- 输出：PDF；
- 租户和数据权限范围；
- 截止时间；
- 是否需要审批。

这里不会偷偷推测关键信息。例如只说“分析 Q1”却没有年份，任务不能直接执行。

### 3.3 生成执行计划

系统建立一个有依赖关系的计划：

```text
知识检索 ───────────────┐
                        ├→ 报告生成 → 独立验证
数据库查询 → 指标计算 ──┘
```

标准计划有四个工具步骤：

1. 检索质量政策和供应商规范；
2. 查询质量数据库；
3. 计算缺陷率与趋势；
4. 生成 PDF 或 JSON 报告。

计划不是随便列出的待办事项，而是一个经过校验的有向无环图。系统会检查：

- 步骤之间是否存在循环；
- 上一步输出能否成为下一步输入；
- 工具是否已经注册；
- 是否超出任务授权范围；
- 重试次数和总时间是否有限。

### 3.4 权限与人工审批

执行工具前，策略模块检查：

- 用户身份；
- 租户；
- 供应商范围；
- 数据分类；
- 工具风险；
- 计划版本；
- 是否存在有效审批。

如果报告包含机密供应商数据，任务进入 `WAITING_APPROVAL`。

审批不是简单的“点一下同意”，而是绑定到：

- 具体计划版本；
- 具体工具；
- 具体供应商和时间范围；
- 审批人角色；
- 有效期；
- 动作指纹。

范围改变后，旧审批不能继续复用。

### 3.5 检索企业知识

`knowledge_search` 负责读取批准的企业知识库，例如：

- 缺陷率定义；
- 供应商质量规范；
- 季度评审规则；
- 企业质量政策。

每个有效片段会形成一条 `DOCUMENT` 证据，记录文档、版本、片段、摘要和校验和。

它不是开放互联网搜索，也不能访问任意知识源。

### 3.6 查询质量数据库

`database_query` 负责查询结构化质量数据。

设计中特别限制：

- 只允许预先登记的查询模板；
- 模型不能直接提交任意 SQL；
- 只允许单条参数化 `SELECT`；
- 必须带租户和供应商过滤；
- 有行数和超时限制；
- 禁止数据库写入、删除、修改结构或调用外部函数。

查询结果形成 `DATABASE` 证据，记录：

- 查询模板；
- 查询指纹；
- 安全的参数摘要；
- 数据快照；
- 行数；
- 数据集校验和。

查询结果为零行不算失败。系统仍会生成报告，但只能说“该范围没有检索到记录”，不能把它解释为“缺陷率为 0%”或“没有质量问题”。

### 3.7 确定性计算

`analysis_engine` 接收数据库数据及其证据，计算：

- 缺陷数量；
- 检验数量；
- 缺陷率；
- 期间趋势。

例如：

```text
S-100：180 / 12,000 = 1.5%
S-200： 80 / 10,000 = 0.8%
```

这些计算由固定代码和公式完成，不交给大模型心算。

计算结果形成 `CALCULATION` 证据，并反向引用数据库证据：

```text
报告中的 1.5%
→ 计算公式和版本
→ 数据库结果
→ 查询模板、参数和数据快照
```

如果分母为零，结果是 `null` 加说明，而不是伪造为 0%。

### 3.8 生成报告

`report_generator` 把以下内容组合成报告：

- 任务范围；
- 指标口径；
- 数据覆盖情况；
- 缺陷率与趋势；
- 主要发现；
- 限制；
- 证据引用。

报告被保存为不可变 `Artifact`，附带：

- 文件位置；
- 类型；
- 大小；
- 校验和；
- 生成器版本；
- 引用的证据 ID。

生成成功只说明“文件已经生成”，不代表整个任务成功。

### 3.9 独立验证

Verifier 会像审稿人一样检查报告：

- 供应商、年份和季度是否与任务合同一致；
- 报告数字是否与计算证据一致；
- 重要结论是否都有正确类型的证据；
- 查询是否只读并符合审批范围；
- 引用能否解析；
- 文件是否存在；
- 文件类型、大小和校验和是否正确；
- 是否写明数据限制和分析边界。

例如报告误把 1.5% 写成 15%，验证器应拒绝报告，并要求在有限次数内重新生成。

只有验证通过，任务才能进入 `COMPLETED`。

### 3.10 返回结果并保留审计链

最终 `TaskResult` 只保存：

- 最终状态；
- 安全摘要；
- 报告 ID；
- 证据 ID。

完整链路是：

```text
TaskRequest
→ TaskContract
→ TaskPlan
→ TaskStep
→ ToolCall
→ ToolResult
→ EvidenceItem
→ StepResult
→ TaskResult
→ Artifact
```

## 4. 任务状态如何变化

任务不是简单的“运行中/完成”，而是有严格状态机：

```text
CREATED
→ UNDERSTANDING
→ PLANNING
→ EXECUTING
→ VERIFYING
→ COMPLETED
```

中间可能进入：

- `WAITING_APPROVAL`：等待审批；
- `RETRYING`：瞬时故障后重试同一步；
- `REPLANNING`：原计划或报告需要修复；
- `FAILED`：不可恢复失败；
- `CANCELLED`：用户取消、审批拒绝或审批过期。

几个重要区别：

- 服务暂时不可用，可以有限重试；
- 数据库零行不是失败；
- 审批拒绝是取消，不是系统故障；
- 权限拒绝不能通过重试或换一种说法绕过；
- 报告内容错误可以有限重生成；
- 完成、失败、取消都是终态，不能让原任务重新复活。

## 5. 各工程模块分别做什么

### 5.1 `api/`：对外服务入口

定位：企业系统通过 HTTP 调用 Copilot 的入口。

计划职责包括创建任务、查询任务状态、提交审批决定、获取报告，以及把内部错误转换成稳定的 API 错误。

**当前实际情况：**只实现了 `/health` 健康检查。任务、审批和报告路由还是空文件。

### 5.2 `cli/`：命令行入口

定位：供开发人员或运维人员从命令行提交、检查任务。

当前只支持接收一段任务文本，以及使用 `--dry-run` 显示“只校验、不执行”。它不会启动智能体、查询数据库或生成报告。

### 5.3 `services/`：业务用例协调层

计划中这是“项目经理”角色。

- `TaskService`：创建并协调任务；
- `ApprovalService`：管理审批；
- `ArtifactService`：管理报告访问。

**当前实际情况：**三个服务文件均为空，是预留边界。

### 5.4 `contracts/`：全系统共同使用的标准表单

这是目前实现最完整的部分。

它定义了：

- 请求长什么样；
- 任务合同包含什么；
- 计划和步骤如何表达；
- 工具如何调用；
- 证据如何记录；
- 审批如何绑定；
- 报告元数据如何表示；
- 错误和状态有哪些合法值。

这些模型是不可变、强类型、拒绝多余字段的。作用类似于企业流程中的标准表单和接口协议，防止各模块各说各话。

### 5.5 `agent/`：智能体的大脑和流程控制器

计划职责包括：

- 理解任务；
- 分类业务类型；
- 生成并校验计划；
- 检查策略；
- 调度工具；
- 汇总证据；
- 组织报告；
- 验证最终结果；
- 根据结果决定下一状态。

这里应该负责“下一步做什么”，但不能直接访问数据库或报告渲染器。

**当前实际情况：**状态、路由、执行图和全部节点文件都是空的，所以完整智能体尚未实现。

### 5.6 `policies/`：权限和风险裁判

计划职责包括：

- 用户权限；
- 数据范围；
- 风险等级；
- 审批要求；
- 访问控制。

每次工具执行前都必须经过这里。

**当前实际情况：**只实现了一个“默认全部拒绝”的安全占位器。在真实策略没有接入前，系统不会误执行工具。

### 5.7 `tools/`：受控能力中心

这一层分为两部分。

#### 通用工具运行框架

已经实现：

- `registry.py`：工具注册表，只有批准的工具名称和风险等级才能注册；
- `executor.py`：统一执行入口；
- `schema.py`：检查工具输入输出是否符合约定；
- `runner.py`：提供有超时限制的执行；
- `base.py`：定义所有工具必须遵守的统一接口；
- `exceptions.py`：把故障转换为安全错误。

一次真实调用的顺序是：

```text
查找工具
→ 校验输入
→ 权限与审批检查
→ 限时执行
→ 校验输出
→ 登记证据
→ 写入审计
→ 返回统一 ToolResult
```

#### 具体业务工具

- `tools/knowledge/`：知识库检索；
- `tools/database/`：只读数据库查询；
- `tools/analytics/`：指标计算；
- `tools/reporting/`：报告和图表生成。

**当前实际情况：**这四个真实工具全部为空。测试里的四个模拟工具仅用于验证通用执行框架，并不连接企业系统。

### 5.8 `evidence/`：证据账本

计划职责包括：

- 保存文档证据；
- 保存数据库证据；
- 保存计算证据；
- 建立证据之间的来源关系；
- 验证引用覆盖。

当前实现了内存版证据账本：

- 证据只追加；
- 自动绑定任务、步骤和工具调用；
- 计算证据必须引用当前任务已有的输入证据；
- 防止跨任务引用证据。

但它还不是可以在服务重启后保留数据的持久化账本。

### 5.9 `persistence/`：任务档案室

计划负责保存：

- 任务；
- 状态和状态事件；
- 计划版本；
- 检查点；
- 审批；
- 报告；
- 审计记录；
- 未来 MCP 会话。

**当前实际情况：**只有工具调用的内存审计仓库实现了，其余任务仓库、Artifact 仓库和检查点等仍为空。

### 5.10 `llm/`：模型适配层

计划负责：

- 定义统一模型接口；
- 隔离 DeepSeek 等具体供应商；
- 管理 Prompt；
- 校验模型的结构化输出。

这样以后更换模型时，不需要改动业务合同、权限或工具代码。

**当前实际情况：**整个模块为空，仓库目前没有实际模型调用。

### 5.11 `observability/`：运行监控

计划负责：

- 结构化日志；
- 调用链追踪；
- 指标；
- 任务、用户和租户关联上下文；
- 延迟、失败率和重试次数。

**当前实际情况：**全为空脚手架。

### 5.12 `config.py`：统一配置入口

这一模块已经实现，用来从环境变量和 `.env` 读取：

- 运行环境；
- 日志级别；
- RAG 服务地址；
- 数据库地址；
- Artifact 目录；
- 最大任务步骤数。

配置错误会被转换成清晰的应用级错误，业务模块不应自行读取零散环境变量。

### 5.13 `mcp/`：未来的协议互操作层

规划中它将让系统：

- 作为 MCP 客户端接入外部批准的能力；
- 作为 MCP 服务端向外暴露明确允许的能力。

它被进一步分成客户端、服务端、传输、安全和协议适配等目录。

**当前实际情况：**MCP 是第五阶段能力，整个目录目前为空；不能把它描述为已经支持 MCP。v1 供应商质量分析也明确把 MCP 排除在范围之外。

### 5.14 `bootstrap/`：未来的系统装配中心

架构要求以后由这里统一创建并连接：

- 数据库适配器；
- 模型客户端；
- 策略引擎；
- 工具注册表；
- 执行器；
- Repository；
- Agent；
- API 服务。

**当前实际情况：**这个目录还不存在，因此现有 API 和 CLI 并没有连接成完整任务系统。

## 6. 辅助工程目录

### 6.1 `tests/`

当前测试主要验证：

- 领域模型字段和不可变性；
- 计划必须是无环图；
- 序列化；
- 工具注册限制；
- 输入输出 Schema；
- 权限拒绝；
- 超时和晚到结果处理；
- 证据登记；
- 审计记录；
- API 健康检查；
- CLI 基础行为；
- 架构依赖规则。

四种业务工具使用的是 `tests/mocks/mock_tools.py` 中的离线模拟对象。

### 6.2 `evaluation/`

最终计划评估：

- 任务成功率；
- 计划质量；
- 工具选择；
- 证据和引用；
- 数字准确性；
- 安全性；
- MCP 互操作。

当前只有一个很小的离线 smoke evaluation，用于验证评估管道能够读取数据、计算 exact match 并生成报告；它不代表产品业务能力已经通过评估。

### 6.3 `scripts/`

当前真正有内容的主要是：

- `run_task.py`：启动最小命令行；
- `check_architecture.py`：检查模块有没有越层依赖；
- `check_docs.py`：检查架构文档和 ADR 是否合规。

运行智能体、初始化数据库、检查任务和 MCP 的脚本仍为空。

### 6.4 `docs/`

这里实际上是目前项目最重要的“产品设计来源”。

`docs/design/design_baseline.md` 冻结了供应商质量分析 v1.0。修改这个场景前必须同时遵守业务范围、领域模型、状态机、工具契约、演练和设计审查六份文档。

### 6.5 `data/`

计划放置：

- 本地演示数据库；
- 生成的报告文件。

当前只有空目录占位文件，没有实际业务数据。

### 6.6 `.github/workflows/ci.yml`

持续集成会检查：

- 代码规范；
- 格式；
- 类型；
- 测试；
- 离线评估；
- 文档治理；
- 架构依赖；
- 包构建。

这说明项目不仅约束“代码能不能跑”，还检查“有没有破坏分层和治理设计”。

## 7. 模块之间的工作方式

可以把整个系统类比为一家受监管的企业分析团队：

| 模块 | 类比角色 |
|---|---|
| API / CLI | 前台接单 |
| Services | 项目经理 |
| Contracts | 标准表单和业务合同 |
| Agent | 工作流调度员 |
| Policies | 权限与合规部门 |
| Tool Registry | 批准能力目录 |
| Tool Executor | 受监管的操作台 |
| Knowledge / Database / Analytics / Reporting | 专业执行人员 |
| Evidence | 证据档案室 |
| Persistence | 业务记录库 |
| Verifier | 独立质检员 |
| Observability | 运行监控中心 |
| LLM | 辅助理解和推理能力 |
| Bootstrap | 系统装配与启动中心 |

最重要的工程原则是：任何角色都不能越权。

例如：

- API 不能直接查数据库；
- Agent 不能直接调用数据库连接；
- 报告工具不能自己再调用分析工具；
- 工具不能修改任务状态；
- LLM 文本不能批准操作；
- MCP 将来也不能建立绕过策略和审计的新通道；
- 生成报告后不能跳过 Verifier 直接宣布成功。

---

## 8. 当前项目到底能做什么

### 8.1 当前已经能够真实运行的能力

- 加载并验证配置；
- 启动 FastAPI 并返回 `/health`；
- 运行最小命令行并接受 dry-run；
- 创建和校验供应商质量分析领域对象；
- 注册符合要求的模拟或插件工具；
- 校验工具输入；
- 在执行前调用授权边界；
- 限时执行同步工具；
- 校验工具输出；
- 登记内存证据；
- 写入内存审计；
- 运行架构、文档、类型和测试质量门禁。

### 8.2 当前还不能真实完成的能力

- 接收任务后自动理解和规划；
- 发起及处理真实审批；
- 连接企业知识库；
- 查询实际质量数据库；
- 计算真实供应商指标；
- 生成真实 PDF 报告；
- 验证完整报告；
- 持久化和恢复完整任务；
- 运行端到端供应商质量分析；
- 使用 DeepSeek 或其他 LLM；
- 使用 MCP。

因此，最准确的项目定位是：

> 一个已经确定业务边界和安全规则、实现了核心契约与工具执行底座，但尚未接通完整业务链路的企业级智能任务执行平台基础工程。
